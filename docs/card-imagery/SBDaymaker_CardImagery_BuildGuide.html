Card Imagery Overhaul:
Your Build Guide

Scenario D, click by click. This page is everything you personally do; the markdown spec (its twin file) is everything Claude Code does. Four phases, four stop-and-shows, two SQL pastes, one settings change, and two cockpit curation sessions. Nothing here requires a terminal.

0Before you start (5 minutes, once)

  1. Put both files in the repo You

    In VS Code's file explorer (left sidebar), find the docs folder. Right-click it, choose New Folder, name it card-imagery. Drag both downloaded files into it: SBDaymaker_CardImagery_BuildSpec.md and this HTML guide (the guide is for you, but keeping it in the repo keeps the record clean).

  2. Open Claude Code You

    Open the Claude Code panel in VS Code the way you normally do. Confirm it shows your sbdaymaker repo as the working folder.

  3. Confirm Supabase access You

    In a browser tab, log into supabase.com, open your SB Daymaker project, and find SQL Editor in the left sidebar. Leave this tab open; you will paste here twice during the build (Phase 2 and Phase 3). You never need it before then.

1Phase 0 · Triage (this week; one session)

What it does: flips the resolver to relevance-first, routes dated events to the clean gradient fallback, adds a repeat-image guard, and re-runs image resolution for the whole catalog. The feed stops being a stock swamp today. No SQL, no settings, fully reversible.

  1. Paste the Phase 0 kickoff prompt You

    Copy the block below into Claude Code and press enter.

    Kickoff prompt · Phase 0 · copy everything belowRead docs/card-imagery/SBDaymaker_CardImagery_BuildSpec.md in full, then read CLAUDE.md. Execute Phase 0 (section 3) only. Begin by reconciling the spec's "current state" section against the live code in ingest/images.ts, ingest/run.ts, ingest/pipeline.ts, and components/ui/Card.tsx; if anything differs, the live code wins and you record the difference in docs/14_SBDaymaker_Build_Deltas.md. Implement the Phase 0 changes, run the tests, then run the forced image backfill through the existing GitHub Actions path (never ask me to run terminal commands). Then stop and show me the Explore feed at about 390px and about 1280px, with the coverage report from spec section 3.1 item 6, and wait for my approval before merging. Do not begin Phase 1.
  2. Review the stop-and-show Wait point You

    Claude Code will show you the feed at both widths plus a coverage report. Check the list below; if anything fails, describe what you see and let it fix before approving.

    • No generic stock photos above the fold (no gyms-in-nowhere, no anonymous coffee cups)
    • Dated event cards show the warm gradient-with-icon look, and it looks intentional, not broken
    • Place cards (restaurants, landmarks) show real photos where they exist
    • No blank cards anywhere; the hero has an image or designed fallback
    • The coverage report lists venues ranked by how many events they host (save this; Phase 2 uses it)
  3. Approve You

    Reply approved, merge. Claude Code merges and confirms. Phase 0 done. Look at the live site on your phone tonight; enjoy it.

2Phase 1 · Real SB photos for places + marquee landmarks (~2 sessions)

What it does: upgrades Wikimedia to location-based search with a strict quality gate, wires food and drink venues to Google (capped), adds photo credits to detail pages, and lets events at the big landmarks (the Bowl, the Lobero, the Courthouse) inherit a beautiful pinned photo. Expected: no SQL needed.

  1. Paste the Phase 1 kickoff prompt You
    Kickoff prompt · Phase 1Continue from the approved Phase 0 of docs/card-imagery/SBDaymaker_CardImagery_BuildSpec.md. Execute Phase 1 (section 4) only. Reconcile against live code first. Before enabling any Google calls, re-verify current Google Places photo pricing and free-tier terms against Google's live pricing pages and record what you find in the Build Deltas ledger; if it is materially worse than the spec's assumption, stop and tell me before any call fires. Build the marquee registry file with empty pins, the geosearch upgrade with the quality gate and unit tests, detail-page attribution, and the food-set Google routing with a scoped backfill. Then stop and show me both widths, including one food venue detail page with its credit line, and wait for approval. Do not begin Phase 2.
  2. Review the stop-and-show Wait point You
    • Landmark-type places show the actual landmark (the Courthouse is the Courthouse)
    • Food venues show their real storefront or interior from Google
    • Open any photo card's detail page: a small credit line appears under or near the photo
    • Nothing ugly slipped the gate (no maps, logos, documents, or weirdly stretched images)
    • Claude Code reports the number of Google calls used; it should be roughly two per food venue, well under 200 total
  3. Approve, then pin your landmarks in the cockpit You (15 to 20 minutes, once)

    After the merge, open your cockpit → Live catalog. For each of the ~12 marquee venues (the Bowl, Arlington, Granada, Lobero, Courthouse, Old Mission, Stearns Wharf, MOXI, Museum of Art, Chase Palm Park, Alameda Park, the Zoo): open its thing, use the photo picker arrows, and pick the shot you would put on a postcard. Options labeled landmark are the curated candidates. When you pick, the cockpit shows a small code snippet; paste that snippet into Claude Code with the words add this pin to the marquee registry. That is the same flow you already use for recurring schedules.

3Phase 2 · Venue photo pools (the big one; ~3 sessions)

What it does: this is your amendment made real. Events inherit real photos of the venue they happen at, drawn from pools of photos you approve in the cockpit, rotated so the feed never repeats, refreshed nightly so Google links never go stale. This phase has your only SQL paste and your only settings change.

  1. Set the spending cap in Vercel You (2 minutes)

    In a browser: vercel.com → log in → open the sbdaymaker project → Settings (top tab) → Environment Variables (left menu) → click Add New. Key: IMAGE_MONTHLY_CALL_CAP · Value: 500 · leave all three environment checkboxes (Production, Preview, Development) checked → Save.

  2. Set the same cap in GitHub You (2 minutes)

    github.com → your sbdaymaker repository → Settings tab → in the left menu, Secrets and variables → Actions → click the Variables tab (not Secrets; this is not sensitive) → New repository variable. Name: IMAGE_MONTHLY_CALL_CAP · Value: 500 → Add variable. If Claude Code later tells you it needs to be a Secret instead because of how the workflow reads it, do the same steps under the Secrets tab.

  3. Paste the Phase 2 kickoff prompt You
    Kickoff prompt · Phase 2Continue from the approved Phase 1 of docs/card-imagery/SBDaymaker_CardImagery_BuildSpec.md. Execute Phase 2 (section 5). Reconcile against live code first. I have set IMAGE_MONTHLY_CALL_CAP=500 in Vercel and GitHub; verify it is being read. Generate the Phase 2 DDL from spec section 5.1, verify it against the live schema, and present it to me as a single copy-paste block, then wait: I will run it in the Supabase SQL Editor and confirm before you build anything that depends on it. Then build the venue seeding, the cockpit Venues tab, pool rotation with per-feed dedupe, and the nightly Google refresh exactly as specified, including the refresh cadence verification in 5.5; if the cap math does not work, bring me the numbers rather than deciding alone. Stop and show me both widths plus the SOhO-style repetition demo from 5.6, and wait for approval. Do not begin Phase 3.
  4. Paste the SQL when asked Wait point You (3 minutes)

    Claude Code will hand you a SQL block (creating venues and venue_photos tables and one new column). In your Supabase tab: SQL Editor → New query → paste the entire block → click Run (bottom right). You should see Success. No rows returned. Go back to Claude Code and type: DDL run, success, no rows returned. If you see any red error instead, paste the error text into Claude Code; it will give you a corrected block. Never edit the SQL yourself.

  5. Review the stop-and-show Wait point You
    • An event at a known venue shows a real photo of that venue
    • The repetition demo: several same-day events at one venue show different photos, and no venue photo appears twice in one feed view
    • The cockpit has a new Venues tab with match review and photo pools, and it feels like the rest of the cockpit
    • Claude Code confirms the nightly refresh ran in a test and reports how many calls it costs per night against the 500 cap, with the math shown
    • Detail pages show credits on venue photos, including the Google photographer name
  6. Approve, then hold your first pool curation session You (60 to 90 minutes, once; the fun part)

    Cockpit → Venues tab. Work top to bottom; the list is ordered by how many events each venue hosts, so the first ten venues you finish cover the biggest slice of the feed.

    • Matches to review: for each proposed match, confirm the event really belongs to that venue. Approve, reject, or reassign. Do this pane first.
    • Photo pools: open a venue, tap Fetch candidates. You will see up to ten Google photos and up to five Wikimedia shots. Approve three to five that pass your bar: would a knowing local friend show this? Reject blurry drinks, random selfies, and anything that misrepresents the room.
    • If every candidate for a venue is bad, approve none. That venue's events simply keep the motif treatment, which is the design working as intended.
    • You do not need to finish in one sitting. Every approved pool starts appearing in the feed after the next nightly run.

    Ongoing: new venues appear in Matches to review as the pipeline meets them; a few minutes during your normal daily cockpit pass covers it.

Two-week checkpoint. After Phase 2 has run for about two weeks, look at the feed like a user. If venue photos are pulling their weight, proceed to Phase 3 on schedule. If any venue's photos feel wrong, unapprove them in the pool; the motif takes over automatically the next night. Nothing in this phase is one-way.

4Phase 3 · The motif tier and goodbye Pexels (~2 sessions + art)

What it does: every card that has not earned a real photo gets a house-drawn v3 ink motif (or the big-type fallback), and Pexels is deleted from the codebase. Second and final SQL paste. The illustrated-identity moat ships here.

  1. Paste the Phase 3 kickoff prompt You
    Kickoff prompt · Phase 3Continue from the approved Phase 2 of docs/card-imagery/SBDaymaker_CardImagery_BuildSpec.md. Execute Phase 3 (section 6). Reconcile against live code first. The working motif art lives in docs/card-imagery/SBDaymaker_Explore_Feed_Mockup_v1.html: port the SVG symbols, filters, and tint classes from that file directly; do not redraw them, and remember that SVG use elements need inline style attributes. Generate the Phase 3 DDL from 6.1, present it as one block, and wait for my confirmation before depending on it. Then build the motif registry, the big-type fallback, the Card.tsx render order, retire Pexels, and run the full forced backfill. Stop and show me a full Explore day at both widths and confirm zero Pexels rows remain.

    Before pasting: drag SBDaymaker_Explore_Feed_Mockup_v1.html into docs/card-imagery/ the same way you did in step 0, so Claude Code can read the art.

  2. Paste the SQL when asked Wait point You

    Same routine as Phase 2: Supabase → SQL Editor → New query → paste → Run → report back DDL run, success. This one adds three columns and one new value to a list; it is even smaller than the last.

  3. Review the stop-and-show Wait point You
    • Scroll a full day: photos where earned, ink motifs everywhere else, and it reads as one designed product
    • No two identical visuals sit next to each other
    • Cards whose category has no motif yet show the big-type treatment (one confident fact in Fraunces), not a hole
    • Claude Code confirms: zero Pexels rows in the database, zero Pexels requests in the nightly log
    • Everything still readable: text on every tint passes contrast (Claude Code reports the check)
  4. Delete the Pexels key You (1 minute, only after the step above passes)

    Vercel → sbdaymaker → Settings → Environment Variables → find PEXELS_API_KEY → the three-dot menu on its row → Remove. Then GitHub → repo Settings → Secrets and variables → Actions → Secrets tab → find PEXELS_API_KEY → remove it there too. Tell Claude Code you did it; it will confirm the next nightly run is clean.

  5. Art sessions You (with me, in chat, ongoing)

    Eight motifs exist; the target library is 24 to 36. We draft the rest together in this chat a few at a time, in the established v3 ink style, and each finished batch drops into the registry with a one-line prompt to Claude Code. Repetition within a category shrinks with every batch.

5If something goes wrong

  • A stop-and-show looks wrong: do not approve. Describe what you see in plain words ("the third card's photo is a map", "two cards show the same bar"). Claude Code fixes and re-shows. Approval is always yours to withhold.
  • SQL error in Supabase: paste the exact red error text into Claude Code. Do not retry with edits of your own.
  • A bad photo reaches the live site: cockpit → Venues → that venue's pool → unapprove the photo. Gone at next render. For a non-pool photo, use the thing's picker arrows to choose a different option, exactly as today.
  • Images look broken or stale: tell Claude Code "check the nightly refresh"; it will read the run logs. The cards are built to fall back to the gradient rather than show a broken image in the meantime.
  • Costs: the 500-call cap is a hard stop enforced in code and you set it yourself in step one of Phase 2. Worst case if every call billed: about $10 in a month. Expected: $0. You can check the meter any time by asking Claude Code for this month's image_spend row.
  • Want it all undone: each phase's merge notes include its revert path. Phase 0 reverts with one line; pools revert by unapproving; motifs revert by render-order change. Nothing destructive ever runs.

6What done looks like

  • Roughly half the feed is real, founder-approved Santa Barbara photography; the rest is the house-drawn identity
  • Zero stock photos, zero repeats within a feed view, zero blank cards
  • Image costs pinned at about $0 with a hard cap you control
  • Every borrowed photo credits its photographer on the detail page
  • Your daily routine unchanged; pool upkeep rides the cockpit pass you already do
The order matters. Phase 0 this week, then 1, then 2, then 3. Each is a prerequisite for the next, each ends in your explicit approval, and the two-week checkpoint after Phase 2 is a real decision point, not a formality.
