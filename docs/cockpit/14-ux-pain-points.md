# Section 15 - Current UX Pain Points (from code inspection)

Every item here is analysis and carries [INFERRED] per the global rule; each cites the code that grounds it and, where possible, the Section 13 task it taxes. Notable negative finding first: there are ZERO TODO/FIXME/HACK comments and no commented-out features anywhere in the scoped code (verified by grep) - the debt below is structural, not flagged-by-the-author.

Ranked by operator-time impact (daily minutes first, aesthetics last):

1. [INFERRED] No morning rollup / no cross-tab badges. Only Queue (count) and Coverage's sub-badges surface work from the shell; Images, Flags, Venues, Edition backlogs are invisible until each tab is opened. The operator's 20-30 minute budget starts with a tab-walk just to discover what exists. Grounding: CMP-01 TABS array has showCount only for Queue; no badge props elsewhere. Task: all of 13.1. This is the highest-leverage fix in the whole Cockpit.
2. [INFERRED] Silent failure on the highest-volume action. Approve/reject are fire-and-forget (ReviewQueue.tsx 49-55): a mid-session auth expiry or 500 shows "Published" and the operator finds the card again tomorrow - re-doing work without knowing why. Task 13.2. Pair any redesign with response checking + a retry toast.
3. [INFERRED] No recovery for a wrong Reject after the 2.6s toast. Rejected items become status archived with no listing screen (only merged rows surface, reviewServer.ts loadMergedRows); recovery = Supabase SQL. One mis-key (R is adjacent to E) can silently delete a good event. Tasks 13.2/13.4. The daily guide even encourages liberal R use ("Use this freely").
4. [INFERRED] Inconsistent undo envelope. Bulk archive has Undo (CatalogView.tsx 305-318); single Delete has only a pre-confirm (167); queue approve/reject have 2.6s Undo; photo applies have per-toast Undo; flags/hero pins/sweep triage have none. The operator cannot build a reliable "can I take it back?" instinct. Tasks 13.3.
5. [INFERRED] Toast Undo is mouse-only and 2.6-3.6s. The span has role="button" tabIndex=0 but no key handler (ReviewQueue.tsx 332, ImagesView.tsx 887, CatalogView.tsx 533), so the keyboard-first operator must grab the mouse within the window. Task 13.2's keyboard coverage gap.
6. [INFERRED] Hidden work caps. Venues "Matches to review" renders only the first 40 proposals with no remainder count (VenuesView.tsx 612); catalog list hard-caps at 2000 rows server-side with a silent empty-200 on failure (API-07 entry); sweep triage shows 50 per pass (documented in-UI, fine) and the images scan caps at 1000 (documented in-UI, fine). The undocumented two hide backlog. Tasks 13.3.
7. [INFERRED] Long browser-resident bulk runs. Images auto-free-all/auto-Google-all run minutes of sequential fetches in the tab with confirm-dialog warnings as the only guard ("keep the tab open", ImagesView.tsx 394-400); no resume, no server-side job. An accidental tab close mid-run leaves a half-processed backlog with no summary of where it stopped. Task 13.3 Images.
8. [INFERRED] Two commit models inside one sheet. The catalog edit sheet mixes instant-apply (photo picker: "Use this photo" writes live immediately) with draft-save (text fields need "Save changes"), visually undifferentiated (CatalogView.tsx 464-527). Cancel discards text edits but not the already-applied photo. Task 13.3 live-catalog fixes.
9. [INFERRED] Per-field save sprawl on the edition desk. Chrome has its own save; each pick has its own save; reorder saves immediately; decision is separate (EditionDraftView.tsx). A full pass over 6-8 picks is 8-10 button presses plus scroll; no save-all, no dirty-state rollup. Task 13.3 edition pass, twice weekly.
10. [INFERRED] Flags failures are invisible and Flags has no badge (items 1 and SCR-11's silent catch, FlagsView.tsx 23-35): a trust-critical loop (visitor corrections) with the weakest feedback in the app.
11. [INFERRED] Session-refresh fragility: no middleware means cookie refresh depends on route-handler traffic (08-auth-permissions.md); combined with item 2, an expired session mid-queue-pass is indistinguishable from success. Needs a live-site verification (open question).
12. [INFERRED] Inconsistent dialog infrastructure: two focus-trapped sheets (catalog, venue detail), five untrapped ones (restock, sources, rhythms, hero picker, swap), three window.confirm flavors, and Escape wired on five screens but not SCR-03/SCR-05/SCR-08. A single sheet primitive would fix a11y and behavior drift at once. Grounding: 04-screens.md accessibility lines per screen.
13. [INFERRED] cockpit.css is a 664-line append-only stylesheet with ~40 hand-inlined rgba() tints, 2 rogue hexes (#C9E0E6), 2 undefined tokens (--rule, --radius-card at line 390), ad-hoc breakpoints, and 96 inline styles across views duplicating the same mono-note pattern (02-design-system.md 3.4-3.5). Zero direct daily minutes, but it is the tax on every future restyle - which is exactly what the downstream AI is about to do.
14. [INFERRED] Vocabulary drift for the same concepts across screens: dismissals are "Not a match" (render-only), "Leave on motif" (permanent), "Looks right as-is" (permanent), "Skip" (render-only); the same tier trio is "Events/Recurring/Places" on SCR-01/SCR-10 but "Tier 1/2/3" on SCR-06 and "T1/T2/T3" on SCR-09. Costs seconds of thought each time, and makes the glossary (Section 16) longer than it should be.
15. [INFERRED] The stale operational doc (Doc 13) still teaches magic-link login and a four-key world; a founder following it today meets password login and eight tabs. Documentation debt, not code debt, but it directly costs onboarding-self time.
