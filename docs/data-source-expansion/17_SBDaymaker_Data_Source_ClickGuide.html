Turning the data sources on

Claude Code writes the adapters. This guide is the handful of steps you do to switch each wave on, watch the events flow in, and make sure every source lands where it should — on Explore, in your recurring registry, or in Discover SB. No coding. The same short rhythm repeats for every wave.

00The rhythm you'll repeat

Every wave is the same five moves. Once you've done Wave 1, the rest is muscle memory.

1 · Turn the wave on→ 2 · Run the nightly job once→ 3 · Check Source Health→ 4 · Approve in the cockpit→ 5 · Paste any rhythms into the registry

Who does what

Throughout this guide: Claude Code = already handled in the build, nothing for you to do. You = a click-step in this guide. Most of your work is steps 3–5: looking at what came in and approving it.

01Before you start (one time)

You almost certainly have all of this already — it's the same access you used to build the site. Just confirm:

  • You can open the project's GitHub repository in a browser and you're an admin (you can see the Settings tab).
  • You can open your cockpit (the review dashboard where events wait for approval).
  • You can open the project in Vercel (so a registry change you commit will deploy).
  • You have VS Code + Claude Code open on the repo — for the one paste step, and to run the build wave by wave.

The only new account you'll create in this whole guide

A free SeatGeek developer key — and not until Wave 4. Everything in Waves 0–3 is keyless. So you can ignore it until you get there.

02Four small skills (learn once, reuse every wave)

These four things come up in every wave. Here they are in full, once — later sections will just say "run the job" or "paste the snippet" and point back here.

Skill A · Run the nightly job by hand

Normally the ingest runs itself overnight. To test a wave immediately, you trigger it manually:

  1. Open the GitHub repository in your browser.
  2. Click the Actions tab along the top.
  3. In the left sidebar, click the nightly ingest workflow (the one Claude Code points you to — likely named something like nightly-ingest).
  4. Click the Run workflow button on the right, then Run workflow again to confirm. It kicks off in a few seconds.
  5. Wait for the green checkmark (usually a couple of minutes), then move to Source Health.
Skill B · Read Source Health in the cockpit

After a run, this tells you whether each new source actually worked.

  1. Open your cockpit and find the Source Health view (the per-source run summary — your labels may differ slightly).
  2. For each source in the wave, read three numbers: did it run, how many qualified (passed the time-gate), and how many dropped (and why).
  3. Healthy looks like: it ran, qualified is greater than zero, and the drops have sensible reasons (e.g. "no clock time", "out of area"). A problem looks like: it didn't run, or qualified is zero with no obvious reason.
  4. If a source looks wrong, jump to "If something's off" (§8) before approving.
Skill C · Approve events into the live site

Qualified events wait in your queue until you say yes. Approving is what puts them on Explore.

  1. In the cockpit, open the review queue.
  2. Skim each event — title, date/time, venue look right? The blurb was written automatically; tweak if you like.
  3. Approve the good ones (they publish to Explore), reject anything off. This is your editorial layer — be choosy.
Skill D · Paste a rhythm into your registry

This is the one genuinely new move. Recurring things — farmers markets, weekly nights, standing tours — don't publish like one-off events. They go into your hand-kept registry file, and the cockpit writes the entry for you.

  1. In the cockpit, open the Registry queue (the tab/filter for recurring items — Claude Code wires this in Wave 1).
  2. Review a rhythm (e.g. "Saturday Farmers' Market"). If it's right, click Approve.
  3. The cockpit shows a small copy button with a ready-made code snippet. Click Copy.
  4. Two ways to place it — pick whichever you prefer:
    Easiest → paste it to Claude Code and say "add this to recurringRegistry.ts"
    Or by hand → open recurringRegistry.ts in VS Code → paste inside the list → save
  5. Commit and let Vercel deploy. The rhythm appears on the site after that deploy — not before. That's expected: nothing recurring goes live until you've placed it.

Why the extra step?

Your registry is deliberately a file you control, not an automatic database. The cockpit does the tedious part (writing the exact entry); you keep the final say and a clean record of every rhythm on the site. It's a few seconds per rhythm, and you only do it once per rhythm — ever.

Skill E · Add a GitHub secret (Wave 4 only)

You'll use this once, to store the SeatGeek key safely.

  1. In the GitHub repository, click Settings.
  2. In the left sidebar:
    Secrets and variables → Actions
  3. Click New repository secret.
  4. Name it exactly SEATGEEK_CLIENT_ID, paste the key into the value box, and click Add secret. Done — the nightly job can now use it.

03The waves, in order

Build and switch on one wave at a time. After each, do a cockpit pass before moving on — that's how you catch a bad source early instead of buried in a flood.

Wave 0

Re-confirm the two rebuilt sources

Claude Code

Visit SB and LiveNotes used to need paid scraping; the build switches them to free routes (a WordPress feed, and a smarter date reader). Nothing to configure — just verify once after Claude Code finishes them:

  1. Run the job (Skill A).
  2. Check Source Health (Skill B) for visitsb and livenotes — both should run and qualify events. Approve a few (Skill C).
Wave 1

Foundation — marquee venues, free feeds, all-cities markets

You

The big one: the Bowl & Lobero, all of UCSB, both libraries, and the five farmers' markets. After Claude Code reports Wave 1 built:

  1. Turn the wave on. If Claude Code added a per-wave switch, flip Wave 1 on (Claude Code tells you exactly where). If not, the sources are already active once merged — skip to the next step.
  2. Run the job (Skill A).
  3. Check Source Health (Skill B) for lobero, sbbowl, ucsb, libraries, farmersMarkets. Each should have run; the first four should qualify dated events; the markets should appear as five items in the Registry queue (not the normal queue).
  4. Approve dated events (Skill C) — they land on Explore.
  5. Place the five markets (Skill D) — approve each, copy its snippet, paste into the registry, commit, deploy. After the deploy they show as recurring rhythms on the site.

Sanity check before Wave 2

Open the live site. A Bowl or Lobero show on a real date should appear on Explore. A farmers' market should appear as a weekly rhythm (only after the registry deploy). If both are true, you're good to continue.

Wave 2

Arts & culture depth + thin-city anchors

You

Granada & Arlington, the museums (MOXI, Natural History, Botanic Garden, SBMA), Music Academy, the Alcazar, Goleta & Carpinteria civic calendars, and the Downtown SB calendar. Same rhythm:

  1. Turn Wave 2 on → Run the job (Skill A) → Check Source Health (Skill B) for each new source.
  2. Approve dated events (Skill C). Watch for two special cases: the museums also add a couple of always-on "spot" entries that belong in Discover SB, and Downtown SB's "1st Thursday" is a monthly rhythm that arrives in the Registry queue.
  3. Place any rhythms (Skill D) — 1st Thursday, and any standing items.
Wave 3

Indie long-tail + curated rhythms

You

Eventbrite (the indie classes/pop-ups/shows nothing else catches), plus recurring nightlife, outdoor operators, the Carpinteria Arts Center, Center Stage, Coastal View, and free nature walks.

  1. Turn Wave 3 on → Run the job → Check Source Health.
  2. Eventbrite will show a lot of drops — that's correct. It's a noisy firehose, and the build filters hard for in-person, local, real-time events. In Source Health you should see plenty of "out of area / online" drops and a clean set of qualified locals. Approve the good ones (Skill C).
  3. Place the rhythms (Skill D): recurring nightlife, the outdoor departures, the Arts Center's monthly faire, standing nature walks.
Wave 4

Optional backstops + the SeatGeek key

You

This wave starts with the one account you need to create.

  1. Get the free SeatGeek key. Go to seatgeek.com/build, sign up (no credit card), create an app, and copy the Client ID.
  2. Store it as a GitHub secret named SEATGEEK_CLIENT_ID (Skill E).
  3. Turn Wave 4 on → Run the job → Check Source Health for seatgeek (and any of allevents / sbcountyArts Claude Code enabled).
  4. Approve the net-new events (Skill C). Many SeatGeek/AllEvents items will already exist from better sources and get merged automatically — that's the dedupe working; you'll only see the genuinely new ones.
  5. Discover SB: the Urban Wine Trail goes in as evergreen content under Discover SB — Claude Code places it; you just confirm it shows there, not in the dated feed.

04"Did it land in the right place?"

The whole point is that each source ends up in the correct surface. Use this to spot-check after each wave:

If it's a…It should appear…How you placed it
One-off dated event (a show, a talk, a museum night)ExploreYou approved it in the normal queue (Skill C)
Recurring rhythm (market, weekly night, standing tour, 1st Thursday, monthly faire)As a recurring entry on the siteYou approved it and pasted the snippet into the registry, then deployed (Skill D)
Always-on place (a winery row, a museum-as-a-place)Discover SBClaude Code placed it; you just confirm it's there

One easy mistake to watch for

If a recurring thing (say a farmers' market) shows up as a single dated event on Explore instead of a rhythm, it was routed wrong. Note it and tell Claude Code — it's a quick fix in that adapter, not something you work around.

05If something's off

  • A source didn't run at all → it may not be enabled, or the build didn't register it. Tell Claude Code which source and what Source Health shows.
  • Qualified is zero, but the source clearly has events → usually the time-gate (events without a clear clock time get dropped on purpose) or the site changed shape. Send Claude Code the source name; it'll check the parser.
  • Source Health notes mention "blocked" or Scrapfly → a site started refusing the free route. This is the rare case the paid fallback exists for. Don't change anything yourself — flag it to Claude Code to decide whether to switch that one source to the fallback.
  • The job mentions "robots" or a skipped path → a site asked not to be crawled on that path. The build respects that and stops rather than working around it. Flag it so you (and your attorney pass) can decide.
  • A rhythm won't show on the site → did the registry change deploy? Recurring items only appear after the commit lands in Vercel.

06Once it's all on: your daily 15 minutes

After the four waves are live, the ongoing job is small and the same every day:

  1. Open the cockpit → the review queue has overnight arrivals.
  2. Approve what's good, reject what isn't (Skill C). This is your editorial voice — the part no aggregator can copy.
  3. Every so often a new rhythm shows in the Registry queue → place it once (Skill D) and you're done with it forever.

That's the model working

The sources do the gathering; the gate keeps out anything without a real time; the cockpit hands you a clean, filtered shortlist. Your judgment on what's actually worth doing is the whole product — this just makes sure you're choosing from everything happening, for free, every day.
