Post-audit update — read alongside the UX-decision layer

This audit remains the gate-review record, but several of its findings were later revised by the 16 Locked UX Decisions + 7 follow-on decisions (rationale in Document 6; integrated into the v4 context, the v2 schema, and the v3 product/architecture docs). Where this audit and those decisions conflict, the decisions govern. The specific revisions:

• Place-first → happenings-first. The audit assumed a flat place-first spine; the product is now organized around a three-tier happenings cascade (discrete-dated → recurring → evergreen), with the evergreen tier as the anti-empty floor. The unified "thing" model is unchanged; a happening_tier classification + recurring_schedules table were added.

• Accessibility / the drum-roll picker (Blocker 5). The audit flagged the need for an accessible alternative to the drum-roll time picker. Resolved by removal — My Plan is now light-by-default and uses a standard native time input; the wizard is an optional path.

• The localStorage "moat" (Finding B8). The audit flagged iOS eviction of on-device saves. Now mitigated by a passwordless magic-link save-restore (the shared_states token store) in addition to the PWA install nudge.

• Map & Happy Hour scope. Both ship light at launch (a Nearby list; a "last confirmed" happy-hour list) with the full Mapbox map and live countdowns deferred to Phase 2 — trimming the heaviest build/maintenance surfaces for the solo operator.

• Personas → occasion tags. The six identity personas became ten occasion/intent tags (thing_tags), demoted from "headline differentiator" to a supporting feature.

• Email & re-engagement. Email is promoted to a first-class 2×/week channel (the proven local-discovery retention engine); web push for installed users is a committed Phase-2 fast-follow. Success metrics (WAU) and the accessibility floor (WCAG 2.2 AA) called for in this audit are carried forward unchanged.

01 — The bottom line

Executive verdict

Read this first. The rest of the document is the evidence behind it.

Verdict

Not yet ready to generate a production-grade build guide — but the gap is specification, not strategy. The product is exceptionally well-designed at the experience level and the strategy is coherent and defensible; the prototype is genuinely impressive. The gating problems are concentrated in four areas, none fatal and most resolvable by decision rather than further research:

1. The data layer and pipeline are labeled "illustrative" — and they are. The schema is missing whole tables and fields the real features need, and the nightly pipeline is a sketch with no failure behavior.

2. ~90% of the product depends on scrapers and a legal posture that aren't built or verified. The facts-not-copy rule solves copyright, but the hiQ v. LinkedIn precedent shows the controlling risk is each site's Terms of Service — which the docs don't address.

3. Several load-bearing platform assumptions are outdated or constrained. The 6-job sequential pipeline can't run on Vercel's free tier as drawn; iOS can evict the localStorage saves that are described as "the moat"; and the cost model rests on a Google credit that ended in February 2025.

4. There are no success metrics, no acceptance criteria, and accessibility is asserted but unverified — which matters because the audience is explicitly older and simplicity-seeking.

The five hardest blockers gating the build guide: (1) harden the data model — add the missing submissions table, the structured happy-hour window, photo URLs, the neighborhood enum, and every jsonb shape; (2) resolve the pipeline-and-hosting architecture, because the design doesn't deploy on Hobby cron; (3) settle the sourcing mechanism and a per-source ToS review — the project's top combined technical-and-legal risk; (4) define success metrics, analytics, and acceptance criteria; (5) commit an accessibility floor with a real contrast audit and an accessible alternative to the drum-roll picker.

Recommended next step is a focused decision-and-specification pass (see §9), not more analysis. Once those decisions are made, a build guide can be both correct and genuinely buildable.

04 — The findings

Severity-ranked findings

Grouped by lens. Each finding names the gap, why it threatens the build, a severity, a clearly-labelled recommendation, and which document to update. Grounded marks findings verified against current external sources (see §11).

🔴 Build-blocker 🟠 High 🟡 Medium 🟢 Low
A

Product & Scope

A1
No success metrics or analytics — and "weekly active user" is undefined
🔴 Blocker
The gap
Phase 1's stated goal is "prove the ritual," but there is no KPI, no instrumentation, and no definition of the ritual working. With no accounts, "weekly active users" — the unit the entire growth and revenue model is built on — is never operationally defined or measured.
Why it blocks
You can't build, or evaluate, toward an undefined target. A build guide that omits analytics ships a product that can't tell you if it's succeeding.
Recommendation
Recommend: choose a privacy-preserving, cookieless analytics tool (e.g. Plausible, or Vercel Web Analytics) to stay consistent with the no-accounts/low-PII posture; define "WAU" operationally (e.g. unique device-days per rolling 7 days); and specify a minimal event taxonomy — app_open, lens_select, save, plan_complete, ticket_handoff.
Update
Business Plan + a new short Metrics & Analytics spec.
A2
No acceptance criteria / definition-of-done for any feature
🔴 Blocker
The gap
The docs describe intent richly but contain no testable "done" conditions for any feature.
Why it blocks
A production-grade build guide is a set of acceptance criteria. Without them the builder invents the boundary of every feature, and "done" becomes a matter of opinion.
Recommendation
Recommend: author acceptance criteria per MVP feature (Given/When/Then or a checklist) before generating the guide — start with the highest-ambiguity features (My Plan, the pipeline, the hero edition selection, the admin cockpit).
Update
A new Acceptance Criteria doc, or inline per-feature in the Product Bible.
A3
Cold-start / content seeding is unplanned — and it's the critical-path bottleneck
🟠 High
The gap
"~100–150 evergreen places" has no launch list, no ordering, no founder-hour estimate, and no minimum-to-launch threshold; the same is true for the 5 guides, First Looks, and seed happy hours. The docs flag the place manager as "heavy at launch" but never scope that load.
Why it blocks
Approving 150 AI-drafted blurbs by hand is the single biggest launch effort, and a daily-ritual app launched with thin content fails its first impression. Without a defined launch gate you can't tell when you're ready to go live.
Recommendation
Recommend: define a launch-content checklist with minimum viable counts (e.g. ≥N places spread across all neighborhoods, all 5 guides live, ≥M confirmed happy hours) as an explicit go-live gate, plus a realistic hour estimate for the seeding pass.
Update
Business Plan / Product Bible.
A4
Empty and error states are largely undefined
🟠 High
The gap
Only the hero fallback is specified. Empty Saved, empty My Plan (no saves), zero live happy hours, a quiet First Looks week, location-denied, offline, and no-search-results are all undefined.
Why it blocks
Empty and failure states are where most real-world sessions diverge from the demo, and they carry the brand voice. Leaving them undefined means the builder ships blank screens or generic errors that contradict the "knowing local friend" tone.
Recommendation
Recommend: define copy and behavior for every empty/error state in a single surface × state matrix (overlaps with C1), written in the product's voice — "an empty screen is an invitation to act," not a dead end.
Update
Product Bible + Wireframe.
A5
First-run / onboarding edge cases unspecified
🟡 Medium
The gap
First-run is detected by a localStorage flag, so cleared storage, incognito, or iOS eviction (see B8) re-onboards the user on every visit; whether the banner ever returns deliberately is undefined.
Why it blocks
Lower severity, but it shapes a returning user's first impression and interacts with the storage-durability issue.
Recommendation
Recommend: decide the re-show policy explicitly and accept (or mitigate) re-onboarding for storage-less sessions.
Update
Product Bible.
A6
The weekend digest is MVP but underspecified — and contradicts "no users table"
🟠 High
The gap
The digest is in MVP scope, but there's no double opt-in, no unsubscribe (legally required under CAN-SPAM), no assembly logic, and no sender-domain authentication (SPF/DKIM/DMARC via Resend). Critically, opt-in emails are real stored PII, which directly contradicts the repeated "deliberate absence: no users table" claim.
Why it blocks
It's both a legal requirement and a schema gap — the architecture has nowhere to store a subscriber, a consent record, or an unsubscribe token.
Recommendation
Recommend: add a subscribers table (email, consent timestamp, unsubscribe token, status), implement one-click unsubscribe, and reframe the "no users table" claim as "no end-user account table" so it's accurate.
Update
Architecture (schema) + Business Plan.
A7
Monetization has no representation in the build
🟡 Medium
The gap
"Featured / sponsored listings" is the highest-fit revenue path, yet there's no schema field, no labeling component, and no placement/ranking rule. The trust guardrail — "sponsored never displaces the genuinely-best pick" — is unenforceable without a defined mechanism.
Why it blocks
Even if revenue is Phase 2, retrofitting sponsorship into ranking later is exactly the kind of rework the project avoided for photos by pre-building the resolver. The same foresight should apply here.
Recommendation
Recommend: decide whether the MVP schema reserves is_featured / sponsor_id fields and a single clearly-labeled placement slot now, so the trust rule is structurally enforceable later.
Update
Architecture (schema) + Product Bible.
B

Architecture & Engineering

B1
The schema is illustrative and incomplete for the real feature set
🔴 Blocker
The gap
The things table and friends are explicitly "illustrative" and, audited against the features, are missing: a submissions table (raw pasted caption + submitter contact PII + consent, before parsing into a draft); ends_at and a structured happy-hour window (day-of-week recurrence + start/end clock times — the prototype's ends: 47 integer is a countdown, not a schedule); a stored photo URL/query for Pexels/Wikimedia (only photo_source exists); is_featured/sponsor fields; hero-eligibility / ranking weight; the guides.stops jsonb shape and whether stops foreign-key to things; editions.pick_ids ordering/cap; a canonical neighborhood enum (the client hard-codes AREA_TO_HOOD); and persona-id integrity (a text[] plus a separate tag_confidence jsonb is fragile and unconstrained).
Why it blocks
The schema is the foundation every other layer sits on. A build guide written against an illustrative schema will need rework the moment happy hours, submissions, or the map are implemented for real.
Recommendation
Recommend: promote the schema from illustrative to specified — add the missing tables and fields above, and define every jsonb shape concretely. This is the single highest-value pre-build artifact.
Update
Architecture.
B2
The nightly pipeline doesn't run on Vercel Hobby cron as drawn — and is otherwise unspecified Grounded
🔴 Blocker
The gap
Vercel's Hobby tier allows cron at a once-per-day cadence with imprecise timing — a job set for a given hour can fire anywhere within that hour. The designed schedule (02:00 ingest → 02:30 enrich → 03:00 stale → 06:00 draft → 06:01 notify → Thu 07:00 digest) has sequential dependencies (ingest must finish before enrich starts) that imprecise within-the-hour timing breaks. Separately, no idempotency, retry, partial-failure handling, or dedupe algorithm is defined, and standard (Hobby) function maxDuration caps at 60s — a multi-site scrape plus a batch Claude call will very likely exceed it (FUNCTION_INVOCATION_TIMEOUT).
Why it blocks
The pipeline is the engine of the whole product, and as drawn it cannot be deployed on the free tier the cost model assumes.
Recommendation
Recommend: pick one — (a) Vercel Pro (~$20/mo, precise scheduling; revise the "$0–40/mo" thesis), (b) a single daily orchestrator cron that runs the steps in sequence internally, with chunking/queueing to stay under the duration cap, or (c) an external scheduler (cron-job.org, GitHub Actions) hitting your API routes. Then specify the dedupe rule (fuzzy match on normalized title + venue + date), the geocoding provider + caching, and per-step failure handling.
Update
Architecture.
B3
The scraping layer — ~90% of the product — is unbuilt and the legal posture is incomplete Grounded
🔴 Blocker
The gap
The non-ticketed content engine depends on scrapers that are neither built nor specified: no tooling choice (headless browser vs. HTML parser), no per-site selectors, no anti-bot or maintenance plan, and no owner for when a venue redesigns its site. Legally, hiQ v. LinkedIn establishes that the CFAA does not bar scraping public data — but a site's Terms of Service is enforceable as breach of contract, plus trespass to chattels (hiQ took a $500K judgment and shut down). So facts-not-copy correctly handles copyright but leaves the controlling ToS/contract risk unaddressed for the named commercial sources (SB Bowl, Granada, Lobero, SOhO, Visit SB, santabarbaraca.com). The civic government calendar is the lowest-risk source.
Why it blocks
This is the product. If the scrapers don't exist and the legal posture on the commercial sources is wrong, the "easy 10%" (licensed APIs) ships and the "90%" that is the actual differentiator doesn't.
Recommendation
Recommend: decide a sourcing mechanism per source — strongly prefer official APIs, RSS, public iCal feeds, and explicit partner permission over scraping; lean on the submission engine for the rest. Commission a per-source ToS review before scraping any commercial site. Treat this as the project's #1 combined technical-and-legal risk, not a footnote.
Update
Architecture + Business Plan (legal).
B4
The AI layer has no concrete contracts
🟠 High
The gap
Prompts exist as snippets, but there's no output schema/validation (JSON mode / tool use), no per-job token/cost budget, no timeout/rate-limit/failure handling, no model-version pinning, and no blurb eval or repetition control. The negative-rule safety layer is described but not specified as code — what's the rule set, where does it run, and what happens when the model violates it?
Why it blocks
The AI is "invisible infrastructure," and infrastructure needs contracts. Without validation and a defined safety rule set, the most trust-sensitive surface (persona tagging) is protected by prose, not code.
Recommendation
Recommend: define structured-output contracts with a validation + fallback step, an explicit negative-rule set implemented as code that runs after generation, model pinning, and a small blurb-quality check. Resolve the My Plan question separately (B5).
Update
Architecture.
B5
My Plan "intelligence": AI vs. deterministic is a genuine, unresolved fork
🟠 High
The gap
The architecture lists "My Plan intelligence" as an on-demand Tier-1 AI job; the wireframe implements it as pure client-side sorting + a random shuffle. computeTravel and the gap-filler ("you've got 90 minutes — Loquita's a 4-min walk") are named/shown but absent from the real logic.
Why it blocks
These are opposite builds. Per-request AI on the one interactive surface contradicts the "batch-only, trivial cost" principle and adds latency to the product's signature moment; the deterministic version is cheaper but loses the "feels like a friend" gap-fill.
Recommendation
Recommend: decide deterministic vs. AI for the pick list explicitly; specify the travel-time source (Mapbox or Google distance) for real; and decide whether gap-fill and the share/ICS exports (currently toasts) ship in MVP or move to a labeled fast-follow.
Update
Architecture + Product Bible.
B6
Image resolver: a proxy contradiction plus quality gaps Grounded
🟠 High
The gap
Good news first: the "$7 per 1,000 after 1,000 free" Place Photo figure is confirmed current (it's the Enterprise "Places Photo" SKU). But the architecture says to proxy Google photos server-side so the key isn't exposed — and proxying every photo through Vercel functions adds invocation cost and latency that can exceed the photo cost itself. There's also no placeholder/blur-up strategy, Pexels category queries risk generic stock ("food" → stock pasta), and Wikimedia attribution rendering is undefined. Note also that the classic Places API is now "Legacy"; new builds should target Places API (New).
Why it blocks
Images are the dominant variable cost and the dominant first impression; the fetch path is load-bearing for both cost and UX, and the current design is internally contradictory.
Recommendation
Recommend: decide the real fetch path (an HTTP-referrer-restricted client key is the lighter option vs. a server proxy), add placeholder + attribution components, define a Pexels query-quality approach, and build against Places API (New) — re-confirming its Place Photo SKU and caching terms at billing setup.
Update
Architecture.
B7
No performance budget, no PWA decision, and push-vs-email is ambiguous Grounded
🟠 High
The gap
"Open it like the weather" makes cold-start the entire UX, yet there's no Core Web Vitals target, no service-worker/offline strategy, no ISR/SSR configuration for the SEO guides the growth thesis depends on, and no image-CDN plan. Whether this is a true PWA or a responsive web app is unresolved. And iOS web push works only for PWAs installed to the Home Screen, triggered by a user gesture — so the founder "edition ready" push requires the founder to install the PWA; email (Resend) is simpler and platform-independent.
Why it blocks
Performance is the product for a daily ritual; the guides' SEO is the inbound channel; and the nudge mechanism changes what gets built.
Recommendation
Recommend: decide PWA vs. responsive; set an explicit CWV budget; configure ISR for guides and an offline shell; and use email for the edition nudge (or specify that the founder installs the PWA).
Update
Architecture.
B8
The "moat" is weaker than stated — iOS can evict the localStorage saves Grounded
🟠 High
The gap
The personal layer ("the moat") rests on saves living in localStorage — but the prototype uses in-memory state, and on iOS Safari can evict localStorage/IndexedDB for non-installed sites after roughly 7 days of inactivity. Because there are no accounts, there is no backup path. The exact "occasional visitor" case the product courts — save on this trip, return next trip — is precisely where eviction bites.
Why it blocks
An architectural assumption (durable on-device storage) is materially weaker on iOS than the docs imply, and it undercuts a feature the strategy calls a moat.
Recommendation
Recommend: decide a durability strategy — nudge Home-Screen install for durability, and/or offer an optional lightweight export/sync — and set expectations honestly in the UI copy. This is also the strongest concrete argument for eventually adding optional accounts.
Update
Architecture + Product Bible.
B9
No per-dependency limits / failure matrix Grounded
🟡 Medium
The gap
No documented free-tier limit, cap behavior, or graceful degradation per third party: Ticketmaster (≈5,000 calls/day, 5 req/s — verify current), Google Places/Geocoding (per-SKU caps post-March-2025; the old $200 umbrella credit is gone), OpenWeather, Mapbox, Resend, Anthropic, and Supabase — whose free tier pauses after 7 days of inactivity and keeps zero backups.
Why it blocks
Each dependency has a different cliff; without a matrix, the app degrades unpredictably and the founder is surprised by both outages and bills.
Recommendation
Recommend: build a one-page dependency matrix (limit / behavior at cap / fallback) and decide free-vs-paid per service. Note that Supabase's pause + zero-backup behavior is a real risk during the labor-intensive seeding phase.
Update
Architecture.
B10
Public forms and admin auth are an unhardened attack surface
🟠 High
The gap
The public submission forms have no captcha or rate-limit (spam), a stored XSS risk via pasted captions later rendered in the cockpit, and an AI prompt-injection vector — a malicious caption fed to the parsing model could try to manipulate it. Admin auth is a single email-and-password account with no 2FA, no login rate-limit, and no defined CSRF posture for server actions.
Why it blocks
Public write endpoints plus an LLM in the loop is a classic injection surface, and a single un-MFA'd admin account is the keys to the whole product.
Recommendation
Recommend: specify input sanitization + output encoding, a captcha + rate-limit on public forms, prompt-injection mitigations (treat captions strictly as untrusted data, never instructions), and admin 2FA + login rate-limiting.
Update
Architecture (security).
B11
Build operability baseline is undefined
🟡 Medium
The gap
No test strategy (unit/integration/e2e), no migration workflow named, no seed scripts, no DB backup/restore plan (free Supabase keeps zero backups), no staging-data strategy, and no CI.
Why it blocks
For a solo-run product, operability is sustainability; the absence of backups and migrations is how a hobby project loses its data and momentum.
Recommendation
Recommend: define a minimum baseline — a migration tool, a seed script, backups (via Supabase Pro or scheduled dumps), and a handful of e2e smoke tests on the critical loop.

Architecture.
B12
The cost model rests on an outdated Google framing and understates reliable-production cost Grounded
🟡 Medium
The gap
The Place Photo unit cost survives, but the model's framing assumes the old world where a $200 monthly credit absorbed mixed Google usage — that credit ended Feb 28, 2025, replaced by per-SKU free caps. And a reliable production deployment realistically needs Vercel Pro (~$20) + Supabase Pro (~$25) ≈ $45/mo in platform fees alone, before photos — versus the "$0–40/mo, mostly free" headline.

C2
Accessibility is asserted but unverified — for an explicitly older audience Grounded
🔴 Blocker
The gap
Accessibility is claimed via "big tap targets," but: there's no WCAG contrast audit (Gold #E0A82E and Sage on the Plaster backgrounds are likely AA failures for normal text); the emoji icon-buttons (the hearts) have no aria-label; there are no visible focus states, no keyboard navigation, and no prefers-reduced-motion handling for the many animations (sun pulse, heart pop, urgency pulse); touch targets are asserted, not measured; and the drum-roll time picker is notoriously inaccessible to screen-reader and motor-impaired users.
Why it blocks
The product deliberately targets an older, simplicity-seeking demographic, which makes accessibility core to the value proposition, not polish — and a public consumer product carries ADA exposure. Building the picker without an accessible fallback bakes in an exclusion of part of the target audience.
Recommendation
Recommend: commit to WCAG 2.2 AA as the floor; run a real contrast audit and adjust the palette where it fails; add aria-labels, focus states, keyboard nav, and reduced-motion to the quality floor; and provide an accessible time-input fallback alongside the drum-roll.

Map-at-scale and drum-roll physics are real, unscoped build tasks
🟠 High

Both are deceptively hard interactions that the prototype fakes (static coords; a basic wheel); a build guide that treats them as trivial will under-budget them.
Recommendation
Recommend: scope both explicitly — map library + clustering strategy, and the picker implementation approach — as named build tasks.
Update
Architecture + Product Bible.
C5
No consolidated design-token handoff
🟠 High
The gap
The palette, type, and motion live only inside the wireframe's CSS. There's no consolidated token source of truth (a Tailwind config or CSS-variable file), no spacing scale, no documented component variants (button / card / sheet), and no motion spec (durations / easings).
Why it blocks
The wireframe CSS is the de-facto spec but it's implicit; a build in Cursor needs an explicit token + component source or every screen drifts.
Recommendation
Recommend: extract a single token source of truth plus a short component and motion spec as a required build input (this can be an expansion of context §17).
Update
A new Design System doc, or an expanded §17.
