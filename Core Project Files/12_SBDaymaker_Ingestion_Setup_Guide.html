Document 12 · One-Time Setup Runbook

Turn on the
nightly engine

Your app is built and live. This guide adds the machine that refills it every night — finding new events, checking their start times, resolving photos — and the review screen where you approve what goes live. You won't write code. Claude Code writes it; you click, paste, and check. Six phases, in order, each ending with a list that tells you exactly where to click to confirm it worked.

⚙️ Setup once, runs nightly 🖱️ No experience assumed 📄 Pairs with Document 13 (daily use)
Before you start — what this costs
  1. Everything here fits inside your existing budget. The nightly engine runs on free tiers (GitHub Actions, the Ticketmaster events feed, your existing Supabase & Vercel). Your only real variable is photos, and that's capped.
  2. Phase 13 sets up three photo sources that were never set up in your first build. Two are free and quick (Pexels, Wikimedia). The third (Google Photos) is the only part of this whole project that uses a credit card — and you'll wire in three independent caps before it can spend a cent. Full click-by-click is in Phase 13.
  3. The photo cap holds the line (~$50/month ceiling). Every card gets a real photo: free sources are tried first, and the paid source fills in automatically only when the free ones come up empty. Past the cap, the rare leftover card uses a placeholder. You'll confirm the exact pricing during Phase 13.
  4. You still need your paid Claude plan for Claude Code (the build tool), same as the first build. Nothing else new to subscribe to.

The build rhythm

Read this once — it's the rhythm of every phase below, the same five moves you already know from the first build.

  1. Open your project in VS Code. Open the VS Code app; it remembers your sb-daymaker folder from last time.
  2. Open the Claude Code panel and paste the phase's 🤖 prompt exactly as written.
  3. When Claude asks you for something (a key, a click on a website), do the 🧑 You step for that phase and paste back what it asked for.
  4. Let it finish, then read the ✅ done when checklist and click where it says to confirm.
  5. Only move to the next phase once every box is checked. If one fails, tell Claude what you saw — it fixes it.
The golden rule of this engine

Two promises are wired in and must never break: (1) a show with a fuzzy time ("around 8") is dropped, never guessed — a wrong time is a locked door. (2) Nothing goes public on its own — every item waits in your review screen until you approve it. Each phase below protects these.

Phase 13

The rest of the sources + photos + nightly email

The remaining local sources are added, photos resolve automatically (free sources first, the paid one only when needed and under the cap), permanently-closed places get archived, and a nightly summary email tells you what's new, what was dropped, and whether any source broke. The engine now runs on a schedule.

Heads up — this phase has the most "you" setup since the very first build

The photo sources were never set up in your first build — they belong here. Two are free and quick; the third (Google Photos) is the only place in this whole project you'll enter a credit card, so we wire in three separate safety caps before it can spend anything. Take this section slowly; it's all click-by-click.

🧑 YOU — set up the three photo sources (do these before pasting the build prompt):

A · Reuse your existing email key (nothing new — 30 seconds)

The nightly summary email uses Resend, which you already set up in your first build (Phase 7). You don't create anything new — just have that same RESEND_API_KEY handy from your credentials file to paste when Claude asks. If you can't find it: resend.com → sign in → API Keys → copy (or create a fresh one).

B · Pexels — free photo source (2 minutes, no card)
  1. Go to pexels.com/api → Get Started and sign up (free).
  2. Once in, it shows Your API Key — click to reveal, then copy it. Keep it for when Claude asks.
  3. That's the whole thing — no billing, no card. Pexels is free for this use.
C · Wikimedia — free photo source (nothing to set up)

Wikimedia (the Wikipedia image library) needs no account and no key — it's an open source Claude can call directly. There's literally nothing for you to do here; it's listed so you know it's covered.

D · Google Photos — the one paid source, from zero (~15 minutes, card required)
⚠️ Read this before you start

This is the only part of SB Daymaker that uses a credit card. Google requires one to turn on the photo service, but you control the spending with three independent caps you'll set below. Most photos come from the free sources above, so the paid one is used sparingly. With the caps in place, a surprise bill is not possible.

Google also gives every account a monthly free credit for this service — for a single-city app, a large share of your paid photo calls may fall inside it. You'll confirm the current pricing and free amount during the build (that's the "audit flag B6" check in the prompt).

D1 — Create your Google Cloud account
  1. Go to cloud.google.com → click Get started for free (or Console if you're already signed into a Google account).
  2. Sign in with any Google account (your Gmail is fine). Accept the terms.
  3. It will ask for a credit card to verify you're not a robot and set up billing. Enter it. (New accounts typically get a free trial credit on top of the always-free monthly photo amount.)
D2 — Create a project called "sb-daymaker"
  1. At the top of the Google Cloud Console, click the project dropdown (says "Select a project") → New Project.
  2. Name it sb-daymaker → Create. Wait a few seconds, then make sure that project is selected in the top dropdown.
D3 — Turn on the photo service (Places API)
  1. In the search bar at the top, type Places API (New) and click it in the results.
  2. Click Enable. (If it asks you to link a billing account, pick the one from D1.)
D4 — Create your key
  1. Search for Credentials (under "APIs & Services") → open it.
  2. Click + Create Credentials → API key. A key string appears — copy it and keep it safe (this is your GOOGLE_PLACES_KEY).
  3. Click Edit API key (or the pencil). Under API restrictions, choose Restrict key and tick Places API (New) only, then Save. (This means even if the key leaked, it could only ever be used for photos — nothing else.)
⚠️ Set all three spending caps now — don't skip these

Cap 1 — A hard daily limit at Google (the real backstop). In the Console search bar, go to APIs & Services → Quotas (or "Quotas & System Limits"). Filter to Places API (New), find the Place Photo request limit, click Edit Quota, and set a low requests-per-day number (Claude will recommend an exact figure during the build, based on the pricing it looks up). Once that many photos are pulled in a day, Google simply stops — it cannot spend past it.

Cap 2 — A billing budget alert (your early-warning email). Search Billing → Budgets & alerts → Create budget. Set the amount to your monthly photo ceiling (around your overall ~$50 image cap) and turn on email alerts at 50%, 90%, and 100%. This emails you if spending climbs — it's your tripwire.

Cap 3 — The app's own counter (Claude builds this next). The build prompt below tells Claude to keep its own running monthly count and stop calling Google before the cap — so the app self-limits before Google's limits even come into play. Three layers, fully independent.

And a one-switch reserve is wired in: if any single source ever starts blocking the engine, you can route just that one through a paid scraper without rebuilding anything.
