01 — The whole system

Architecture overview updated

A two-sided web application on a deliberately lean, solo-operable stack: a Next.js app on Vercel Pro, a Supabase (Postgres) data layer, a Claude-powered editorial engine, and a small set of licensed external data/media APIs. One person can run it in minutes a day.

LayerWhat it holds
ClientConsumer app (Next.js / React, installable PWA, mobile-first) · Admin cockpit (desktop + mobile)
Application / edgeNext.js API routes & server actions · Vercel Cron (Pro) · the image resolver
DataSupabase Postgres (things with happening-tier, thing_tags, recurring_schedules, happy_hour_windows, guides, editions, submissions, subscribers, shared_states, audit) · Supabase Auth (admin + 2FA) · Supabase Storage (operator photos, later) · device storage (two-state saved list)
IntelligenceClaude API — blurbs · cascade ranking · category + occasion tagging (the live app is deterministic, not here)

Every choice optimizes for one-person operability and low fixed cost: managed services over self-hosted infra, batch jobs over real-time, on-device state over accounts, and AI-drafts-you-approve over manual production.

03 — The unified model

The data architecture updated

The heart is the unified "thing" model — places, events, First Looks, and happy hours in one table with a type discriminator — now classified by a happening tier that drives the whole-app cascade, so the hero, feed, Near Me sort, Discover SB guides, and saved list all draw from one pool, in one order. The schema is specified and runnable in sbdaymaker_schema.sql, which supersedes the earlier illustrative sketch.

The happenings-first cascade — the spine

Every thing carries a stored happening_tier (1/2/3) + a happening_category (one of 16) + a reason_to_go. Tier is derived from structure, not AI-judged: has a specific date → Tier 1 (discrete, dated — leads, owns the hero); carries a recurring_schedules row → Tier 2 (recurring rhythm, on its scheduled days); neither → Tier 3 (evergreen, activity-framed — the anti-empty floor). The AI proposes only the category.

Lead-and-blend (deterministic): Tier-1 leads (cap ~4) → Tier-2 active-today fills → Tier-3 always backfills to a minimum feed length (~12), so the screen is full on quiet and busy nights. Two negative-rule gates (code): Tier-1 with no date → block + route to Review; Tier-3 with no reason_to_go → block + route to Review ("no bare places"). Horizons (This Week / This Month) are time-filters over this one model, not separate systems.

Identity model — no accounts, one PII boundary

There is no end-user account table — saves live on-device (two-state: "want to go" / "been"). Stored end-user PII is limited to the user's own email — digest opt-in (subscribers) and the optional magic-link save-restore delivery address (shared_states) — treated as one boundary. Saved-list share-links store no recipient contact info (shared via the native share sheet). Public reads are limited to status = 'published' via RLS; submissions/subscribers/sponsors/audit/shared_states are service-role only.

04 — Where content comes from

Data sources & ingestion updated

Ticketed APIs are the easy ~10%; the other ~90% comes from a portfolio of sources. All ingestion pulls facts — never source marketing copy. Inbound from the public forms lands in the submissions table first, then is parsed and approved into a draft thing.

Sourcing posture — scrape facts, with guardrails

The decision is to scrape facts, made defensible by guardrails (because hiQ v. LinkedIn established that a site's Terms of Service — not copyright — is the controlling risk):

GuardrailWhy
Check robots.txt + ToS per source firstCivic gov calendar is lowest-risk; commercial venue/tourism sites are higher-risk
Never create an account or click through a ToSThat's what converts public scraping into breach of contract
Prefer official API / iCal / RSS feeds firstSame facts, explicitly published — far lower risk than scraping a page
Good-citizen scraping (honest UA, rate limits, off-peak)Lowers trespass-to-chattels exposure
Per-source adapters that fail gracefully + alertSubmission engine + manual curation are the fallback when one breaks
Pre-launch IP/tech attorney reviewCheap, high-value de-risking of the whole posture

The copyright-clean guarantee still holds architecturally: the pipeline stores facts only; the blurb column is populated solely by the AI writing original prose. Direct FB/IG scraping remains out of scope.

05 — Photos, legally & cheaply

The image pipeline updated

A resolver picks the cheapest legal source per context. The big correction: Google photos are fetched with a referrer-restricted client key — not a server proxy.

// resolver order — cheapest legal source wins
function resolvePhoto(thing){
  if (thing.photo_source === 'owned')  return ownedUrl(thing);      // Phase 2
  if (isAmbient(thing))   return pexels(thing.photo_query);   // free, curated
  if (isLandmark(thing))  return wikimedia(thing.title);      // free, CC, attributed
  if (thing.place_id)     return googlePhoto(thing.place_id);   // referrer-keyed, lazy
  return placeholderGradient(thing.type);                      // branded fallback
}

Why a referrer-restricted key, not a proxy: Maps keys are designed for client-side use locked to a domain, so a leaked key is useless elsewhere — and we avoid paying a Vercel function invocation + bandwidth on every photo (which could exceed the photo cost itself).

Safeguards (all in the build)

Lazy-load every image (the cost lever — Google photos can't be cached); a hard Google billing cap + alert (~$50/mo); resolution order (free sources first); branded placeholder + blur-up; Wikimedia attribution rendered on landmark images. Build on Places API (New); re-confirm the Place Photo SKU ($7/1k after 1k free/mo, confirmed) + caching terms at billing setup.

06 — The editorial engine

The AI layer updated

The Claude API is invisible infrastructure serving curation. It runs three nightly jobs — the live app is not one of them; every section is deterministic (see §09).

JobOutputValidationModel
Editorial blurbsplain textlength + no-copied-phrasing; regenerate on failHaiku
Category + occasion taggingstructured (tool use)enum-valid category + negative-rules + confidence threshold (tier is derived, not tagged)Haiku
Cascade / hero rankingstructuredids exist + hero_eligible + Tier-1-leads + lead-and-blend varietySonnet
Submission parsestructuredschema + sanitize (untrusted input)Haiku

The negative-rule layer — code, not prose

Runs after the model, before any write: is_21_plus → strip 'family_day' · price_band ≠ free → strip 'free_sb' · happening_tier = 1 && no date → block + route to Review · happening_tier = 3 && no reason_to_go → block + route to Review · confidence < 0.6 → don't publish, route to Content Review · tag_source = founder → overrides. Model IDs are pinned (never "latest"); every call has a timeout + 1 retry + fallback (skip & flag; hero → evergreen). Cost is cents/day at this batch size.

07 — The nightly pipeline

Background jobs & automation updated

A single orchestrator function on Vercel Pro runs the steps in sequence (race-free), with Fluid Compute for duration headroom. This replaces the earlier sketch of six independently-scheduled crons, which couldn't guarantee ordering on any tier.

// one orchestrated nightly run (early morning)
ingest()        // TM API + venue/civic facts (guardrailed) + inbound submissions
  .then(normalize)  // → unified things, dedupe (fuzzy title+venue+date), geocode + cache
  .then(enrich)     // Claude blurbs + category + occasion tags; tier DERIVED from structure; negative rules + the 2 tier-gates applied
  .then(staleCheck) // flag/drop happy hours + recurring + places past last_confirmed
  .then(draftEdition) // Claude cascade-ranks → hero (best Tier-1) + lead-and-blend feed; weather-aware
  .then(notify);    // email "edition ready"; auto-publishes by veto if untouched

// 2×/week email cron — assembles from the ALREADY-APPROVED cascade (no net-new editorial)
wednesday(() => assembleMidweek().then(sendWithVeto));   // "What's On": tonight + out-this-week + just-opened
friday(()    => assembleWeekend().then(sendWithVeto));   // "The Local's Weekend": marquee + week-ahead + evergreen + a guide

// daily reaper — sliding-expiry sweep of shared_states (idle plan + save-restore tokens)
daily(() => reapExpiredSharedStates());

Graceful degradation throughout — if a step fails, it's skipped and flagged, and the hero falls back to the evergreen "perfect SB day" card. The ritual is never blank.

09 — On-device, no accounts

The personal layer & sharing updated

The save → share loop is the moat — the half no calendar offers — and it runs entirely client-side and fully deterministic — no per-request AI.

Save durability — the honest caveat

Saves persist in device localStorage, each with one of two states — "want to go" (default) or "been." iOS Safari can evict storage for non-installed sites after ~7 idle days, and there are no accounts to restore from. Mitigations: the app is an installable PWA that nudges home-screen install (durable storage + the Phase-2 push channel), plus a passwordless magic-link save-restore — "email me a link to restore my saves" writes a one-off snapshot to shared_states (kind=save_restore) keyed by an opaque token and emailed to the user's own address; opening it restores saves on any device. No password, no login wall. Full accounts stay deferred.

Near Me — an in-view sort (no map, no Mapbox in V1)

Near Me is a sort available on Explore and Saved, not a screen. Geolocation first → on grant, the user's neighborhood floats to the top; on deny/unavailable, default to an SB center (downtown/State St.) and a one-tap neighborhood picker. It reads the stored nearby_zone on each thing (no client-side AREA_TO_HOOD, no map tiles). The full Mapbox map is Phase 2.

Sharing — the view-only saved-list link

From Saved, the user shares one item or a multi-select batch. The surviving social artifact is a saved-list view-link — deterministic, client-driven, no AI:

// Share a saved list (no AI) — pick one or many, then share
list = saved.filter(s => selected.has(s.id))   // one item or a multi-select batch
token = buildSharedList(list)   // → shared_states(kind=shared_list); payload = thing ids

// One Perfect SB Day — a deterministic, hand-curated lineup
perfectDay() => seedSaved(['x2','p3','p1','p2'])  // seeds the saved list; reuses existing things

// Shareable saved-list link — viewable + copy-to-tweak, NO account to view
shareList(token) => navigator.share(listUrl(token))  // native share sheet; no recipient PII stored

The link writes a list of thing ids to shared_states (kind=shared_list) under an opaque token; anyone can open it to view the picks and save their own copy (no real-time co-edit — that needs accounts). Delivery is the OS share sheet, so the user picks iMessage/WhatsApp/email and SB Daymaker stores no friend's contact info and pays no send cost. One Perfect SB Day is a one-tap, hand-curated morning-to-sunset lineup on Explore that seeds the saved list to tweak and share. (Retired in v9: the timed itinerary, the .ics export, the haversine travel estimate, and the drum-roll picker — all part of the removed My Plan builder.)

10 — The guardrails

Security, privacy & cost updated

Security

Supabase Auth + RLS (public reads published rows only); secret keys (TM, Claude, weather, Supabase service role) server-side only; the Google Maps key is client-side but referrer-restricted. Admin: 2FA + login rate-limiting. Public forms: captcha + rate-limit, input sanitization + output encoding, and pasted captions treated strictly as untrusted data (prompt-injection mitigation). CSRF posture defined for server actions.

Privacy & accessibility

Near-zero personal data: no accounts, saves on-device. The only stored PII is the user's own email — digest opt-in and the optional magic-link save-restore address — treated as one boundary (consent + unsubscribe; opaque tokens; sliding expiry; a light privacy review at build time confirms token entropy + that the reaper deletes). Saved-list share-links store no recipient PII. Analytics is cookieless. WCAG 2.2 AA is the build floor — the palette usage rule keeps text contrast compliant; aria-labels, focus, keyboard nav, and reduced-motion are in the quality floor.

Cost (re-derived)

The old "$0–40/mo" assumed free tiers + the Google $200 umbrella credit, which ended Feb 28 2025. Reliable production ≈ ~$45/mo platform (Vercel Pro + Supabase Pro) before metered photos ($7/1k after 1k free, the dominant variable). All-in ≈ $93/mo at 100 weekly users, ~$300 at 500, ~$557 at 1,000 — before the operator-photo program bends photos down. Still remarkably cheap; the framing is now honest. (Full table: Document 5 §04.)

12 — End-to-end trace

A day in the data updated

Following tonight's hero pick from source to screen.

  • Early AM — Orchestrate. The nightly function ingests the TM API + the Bowl's calendar facts (guardrailed), normalizes "Trombone Shorty, June 19, 6:30 PM" to a things row, dedupes, geocodes (cached).
  • Enrich. Claude (Haiku) drafts the blurb, sets category live_music and tags catch_a_show, date_night with confidence; tier is derived as 1 (it has a date); the negative-rule layer + the two tier-gates run; it's written status: draft.
  • Rank. Claude (Sonnet) cascade-ranks — this Tier-1 thing leads and takes the hero; Tier-2/3 backfill the feed (lead-and-blend); it writes a draft edition + ordered edition_picks.
  • Notify. You get an email: "Tonight's edition is ready." You approve in the cockpit (or it auto-publishes by veto).
  • Serve. A user opens the PWA. The app reads the published edition; the hero renders; the Bowl photo lazy-loads from Google via the referrer-restricted key only when scrolled into view.
  • Personal. They heart it → spring-pop → saved to device storage as want_to_go. A "Share your list →" nudge appears. From Saved they pick one or several and share a real saved-list view-link (written to shared_states, kind=shared_list) via the native share sheet — a friend opens it, sees the picks, saves their own copy. One Perfect SB Day can seed the list with a ready-made lineup first.
  • Hand-off. "Get tickets" opens the AXS URL. SB Daymaker never touches the transaction.
